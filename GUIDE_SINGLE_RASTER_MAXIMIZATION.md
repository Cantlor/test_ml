# Как максимально использовать один растр перед добавлением нового

Дата: 2026-03-11

Этот документ про текущую конфигурацию проекта (без изменения скриптов), с фокусом на улучшение обучения только на `first_raster`.

## 1) Делить текущий растр на части или сначала добывать новый?

Короткий ответ:
1. Сначала максимально выжать текущий растр.
2. Физически резать его на файлы не обязательно.
3. Полезнее сделать "логические" пространственные зоны (train/val/test по географии), чтобы честнее мерить обобщение.

Почему:
- У вас уже patch-based пайплайн, и модель видит много локальных окон из одного источника.
- Главный текущий bottleneck не количество файлов, а качество boundary-обучения и покрытие сложных негативов.
- Новый растр нужен позже для роста доменного разнообразия, но сначала стоит устранить узкие места в текущем датасете.

## 2) Что сейчас ограничивает качество

По последнему отчёту:
1. `extent` учится хорошо (`test IoU ~ 0.88`).
2. `boundary_f1` почти нулевой (основная проблема).
3. В `prep` есть сильный недобор негативов (`45` вместо `120` целевых).

Следствие: модель хорошо выделяет площадь полей, но плохо стабилизирует границы.

## 3) Что за что отвечает в конфигах

## 3.1 `module_prep_data/prep_config.yaml`

1. `patching.target_patches_per_dataset`
- За что отвечает: общий объём обучающих патчей на датасет.
- Если увеличить: больше разнообразия, дольше подготовка/обучение.

2. `patching.sampling.weights.center / boundary`
- За что отвечает: баланс позитивных патчей (центр поля vs зона границы).
- Если поднять `boundary`: больше примеров для boundary-головы.

3. `patching.sampling.negatives.ratio`
- За что отвечает: доля негативных патчей.
- Если увеличить: лучше отделение фон/граница, но можно "забить" датасет пустыми участками.

4. `patching.sampling.negatives.min_distance_to_fields_m`
- За что отвечает: насколько далеко негативы от полей.
- Если уменьшить: легче набрать нужное число негативов (у вас сейчас недобор).

5. `patching.filters.neg_max_mask_ratio`
- За что отвечает: сколько допустимо маски в негативном патче.
- Если чуть увеличить: уменьшится отбраковка негативов.

6. `patching.filters.min_valid_ratio`
- За что отвечает: минимально допустимая доля валидных пикселей.
- Если уменьшить: добавите near-NoData примеры, но может вырасти шум.

7. `labels.ignore_zone.ignore_radius_px`
- За что отвечает: ширина игнорируемой зоны вокруг границ для `extent_ig`.
- Слишком большой радиус -> ослабление supervision возле границы.

8. `split.ratios.*`
- За что отвечает: размер train/val/test.
- Для 1 растра важно держать достаточно test/val для честной оценки.

## 3.2 `module_net_train/configs/train_config.yaml`

1. `train.epochs`
- За что отвечает: длительность обучения.
- Если увеличить: может вырасти качество, если нет переобучения.

2. `loss.weights.extent / boundary`
- За что отвечает: приоритет задач в multitask.
- Сейчас главный рычаг для boundary — повышать `boundary` вес.

3. `metrics.boundary.thresholds`
- За что отвечает: на каких порогах смотреть boundary F1.
- Больше порогов = лучше диагностика реальной рабочей точки.

4. `train.checkpoint.monitor`
- За что отвечает: по какой метрике выбирать `best.pt`.
- Если мониторить только extent, boundary может не улучшаться даже при хорошем чекпоинте.

5. `model.base_channels` (и архитектурные параметры)
- За что отвечает: ёмкость модели.
- Больше ёмкость = потенциально лучше границы, но выше требования к ресурсам.

6. `sampling.crop_size`
- За что отвечает: размер обучающего кропа.
- Больше кроп = больше контекста для сложных границ.

7. `normalization.p_low/p_high`
- За что отвечает: робастная нормализация каналов.
- Влияет на стабильность и контраст признаков.

## 4) Практический план улучшения (только на текущем растре)

Делайте по шагам, не всё сразу.

## Шаг 1. Закрыть недобор негативов (самый дешёвый буст для boundary)

В `prep_config.yaml`:
1. `patching.sampling.negatives.min_distance_to_fields_m`: `15 -> 8..10`
2. `patching.filters.neg_max_mask_ratio`: `0.01 -> 0.02..0.04`
3. `patching.sampling.negatives.ratio`: `0.15 -> 0.20..0.25`

Цель шага:
- чтобы фактически записанных негативов было близко к целевым.

## Шаг 2. Увеличить обучающий объём и сложность boundary

В `prep_config.yaml`:
1. `patching.target_patches_per_dataset`: `800 -> 1200..2000`
2. `patching.sampling.weights`:
- было `center=0.6`, `boundary=0.4`
- попробовать `center=0.5`, `boundary=0.5`

Цель шага:
- усилить покрытие границ без добавления нового растра.

## Шаг 3. Перенастроить обучение под boundary

В `train_config.yaml`:
1. `loss.weights.boundary`: `3.0 -> 5.0..8.0`
2. `metrics.boundary.thresholds`: добавить сетку, например `[0.15, 0.25, 0.35, 0.50]`
3. `train.epochs`: `20 -> 40..80`
4. `train.checkpoint.monitor`:
- для boundary-эксперимента поставить `val/boundary_f1@0.35`
- либо перейти на комбинированный критерий (если позже захотите доработать логику).

Цель шага:
- перестать оптимизировать только extent и начать реально двигать boundary-метрики.

## Шаг 4. Если есть ресурс GPU/CPU — добавить контекст и ёмкость

В `train_config.yaml` и `hardware_config.yaml`:
1. `sampling.crop_size`: `256 -> 384` (если влезает память)
2. `model.base_channels`: `16 -> 24/32`

Цель шага:
- лучшая локализация сложных контуров.

## 5) Какие метрики считать ключевыми прямо сейчас

Приоритет мониторинга:
1. `test extent_iou` (главная производственная метрика сегментации).
2. `test boundary_f1@0.35` и `@0.50` (главный текущий risk).
3. gap `val -> test` по extent и boundary (стабильность обобщения).
4. из `patches_all manifest`: фактический shortfall по негативам.

## 6) Когда уже подключать новый растр

Подключать второй растр имеет смысл, когда:
1. Вы закрыли недобор негативов.
2. На текущем растре boundary F1 стабильно растёт и упирается в плато.
3. Дальнейшие улучшения на одном растре дают минимальный прирост.

Тогда новый растр даст главный эффект: расширение доменного разнообразия (сезонность, фон, шумы, текстуры).

## 7) Рекомендуемый порядок экспериментов (минимум итераций)

1. Эксперимент A (данные): только Шаг 1.
2. Эксперимент B (данные): Шаг 1 + Шаг 2.
3. Эксперимент C (обучение): Шаг 3 на лучшем наборе из A/B.
4. Эксперимент D (ресурсный): Шаг 4, если доступна память.

Фиксируйте для каждого прогона:
- конфиг,
- лучшую эпоху,
- `test extent_iou`, `test boundary_f1@0.35/@0.50`,
- shortfall негативов.

Так вы получите управляемый прогресс без изменения скриптов и без добавления нового растра на первом этапе.
